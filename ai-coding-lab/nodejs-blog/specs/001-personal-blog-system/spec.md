# Feature Specification: 个人博客系统 (Personal Blog System)

**Feature Branch**: `001-personal-blog-system`

**Created**: 2026-06-24

**Status**: Draft

**Input**: User description: "构建一个功能完整的个人博客系统，包含用户系统（GitHub OAuth/邮箱登录、三级角色）、文章系统（Markdown编辑、标签分类、SEO）、前后台页面（首页/详情/分类/标签/关于、Dashboard/内容管理）、ISR性能优化、图片上传"

## Clarifications

### Session 2026-06-24

- Q: 当用户先用邮箱注册，后用 GitHub OAuth 登录且 GitHub 主邮箱与已注册邮箱相同时，系统如何处理？ → A: 自动关联：通过邮箱匹配自动合并账号，GitHub 登录后进入已有账号（同时绑定 GitHub ID）。
- Q: 文章的 URL 路径应如何生成？ → A: 混合模式 `/posts/[id]-[slug]`，ID 保证唯一性，slug 由标题自动生成（中文转拼音），作者可手动编辑。
- Q: 哪些角色可以上传图片？ → A: 作者和管理员可上传（作者上传自己的图片，管理员可管理所有图片）。
- Q: 关于页面的内容应如何管理？ → A: 管理员可在后台编辑站点设置（关于页内容、博主名称、社交链接等），前台渲染为静态页面。
- Q: 仪表盘统计数据应该实时计算还是缓存展示？ → A: 定时缓存：每 5 分钟刷新一次统计数据缓存，仪表盘读取缓存值。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 文章浏览与发现 (Priority: P1) 🎯 MVP

读者通过首页浏览最新发布的文章，可按分类或标签筛选感兴趣的内容，
点击文章进入详情页阅读完整的 Markdown 渲染内容，查看文章作者、
发布时间、阅读量和标签信息。读者还可以通过关键词搜索文章。

**Why this priority**: 博客的核心价值在于内容消费。没有可浏览的文章，
其他所有功能（写作、管理）都没有意义。这是面向最终用户的核心体验。

**Independent Test**: 部署系统并发布至少一篇测试文章后，访问首页即可
查看文章列表，点击进入详情页阅读完整内容，通过标签和分类筛选文章，
使用搜索框查找特定文章。无需登录即可完成全部测试。

**Acceptance Scenarios**:

1. **Given** 系统中有多篇已发布文章，**When** 读者访问首页，**Then** 按发布时间倒序展示文章列表，每页最多 20 篇文章，底部显示分页导航。
2. **Given** 读者在首页浏览，**When** 点击某篇文章标题或封面图，**Then** 进入文章详情页，展示 Markdown 渲染后的内容、作者昵称、发布时间、标签列表、阅读量计数。
3. **Given** 读者在文章详情页，**When** 点击某个标签，**Then** 跳转到标签页，展示所有包含该标签的已发布文章列表。
4. **Given** 读者在首页或导航栏，**When** 选择一个分类，**Then** 跳转到分类页，展示该分类下的所有已发布文章列表。
5. **Given** 读者使用搜索功能，**When** 输入关键词并提交，**Then** 返回标题或内容匹配的已发布文章列表。
6. **Given** 任意页面，**When** 读者在移动设备上访问，**Then** 页面布局自适应屏幕宽度，导航菜单折叠为汉堡菜单。
7. **Given** 读者访问关于页面，**When** 页面加载完成，**Then** 展示博主个人介绍等静态内容。

### User Story 2 - 用户注册与登录 (Priority: P2)

访客可以通过邮箱和密码注册账号，注册后默认为读者角色。已注册用户
可以使用邮箱密码登录，也可以使用 GitHub 账号一键登录。登录后系统
根据角色展示对应的功能入口。

**Why this priority**: 用户系统是内容创作和管理的基础。必须先有身份认证
才能区分作者和管理员，实现基于角色的权限控制。排在 P2 是因为读者
浏览内容不需要登录，但写作和管理必须登录。

**Independent Test**: 访问注册页面，使用邮箱和密码创建新账号，确认注册
成功后自动登录。退出后使用相同凭据登录。使用 GitHub 账号进行 OAuth
登录。验证不同角色登录后看到不同的功能入口。

**Acceptance Scenarios**:

1. **Given** 未登录的访客，**When** 访问注册页面并填写邮箱、密码、昵称后提交，**Then** 账号创建成功，自动登录并跳转到首页，默认角色为读者。
2. **Given** 已注册用户，**When** 在登录页面输入正确的邮箱和密码后提交，**Then** 登录成功并跳转到首页。
3. **Given** 已注册用户，**When** 在登录页面输入错误的密码，**Then** 提示"邮箱或密码错误"，不跳转页面。
4. **Given** 拥有 GitHub 账号的访客，**When** 点击"GitHub 登录"按钮并授权，**Then** 系统创建或关联账号，自动登录并跳转到首页。
5. **Given** 已登录的用户，**When** 点击退出登录，**Then** 清除登录状态，跳转到首页，不再显示需要登录的功能入口。
6. **Given** 读者角色用户登录后，**When** 浏览页面，**Then** 不显示后台管理入口和文章写作入口。

### User Story 3 - 内容创作与管理 (Priority: P3)

作者登录后进入后台文章管理页面，可以创建新文章、编辑已有文章、
删除自己的文章。创建和编辑文章时使用 Markdown 编辑器并支持
实时预览。作者可以为文章设置标题、摘要、封面图、标签、分类、
SEO 元信息和发布状态（草稿/发布/归档）。

**Why this priority**: 内容是博客的生命线。作者工具链完善程度直接影响
内容产出效率和质量。排在 P3 是因为必须先有内容消费和用户系统，
作者才能开始写作。

**Independent Test**: 作者登录后进入文章管理页面，点击新建文章，
在编辑器中输入 Markdown 内容并观察预览，设置标签和分类，
上传封面图，填写 SEO 信息，保存为草稿。确认草稿仅作者本人可见。
将草稿发布，确认读者可在首页看到。编辑已发布文章并保存，
确认更新生效。将文章归档，确认文章从首页列表移除但详情页仍可访问。

**Acceptance Scenarios**:

1. **Given** 作者已登录，**When** 访问后台文章管理页面，**Then** 以列表形式展示自己创建的所有文章（含草稿、已发布、已归档），每篇文章显示标题、状态、发布时间、阅读量，支持状态筛选。
2. **Given** 作者在文章管理页面，**When** 点击"新建文章"，**Then** 进入编辑器页面，左侧为 Markdown 编辑区，右侧为实时渲染预览区。
3. **Given** 作者在编辑器中，**When** 输入 Markdown 内容（含标题、代码块、图片、列表），**Then** 预览区实时更新渲染结果，支持语法高亮。
4. **Given** 作者在编辑器中，**When** 填写标题、摘要、选择 1-5 个标签、选择 1 个分类、上传封面图、设置 SEO 标题/描述/OG 图片，选择状态为"已发布"后保存，**Then** 文章发布成功，读者可在首页看到。
5. **Given** 作者保存了一篇草稿，**When** 在文章列表中将其状态切换为"已发布"，**Then** 文章立即对读者可见。
6. **Given** 作者已发布一篇文章，**When** 编辑该文章并修改内容后保存，**Then** 文章内容更新，修改立即生效。
7. **Given** 作者拥有一篇文章，**When** 点击删除并确认，**Then** 文章及其关联数据被移除，不再出现在任何列表中。
8. **Given** 作者 A 已登录，**When** 尝试通过直接 URL 访问作者 B 的文章编辑页面，**Then** 系统拒绝访问并提示"无权操作"。

### User Story 4 - 系统管理与数据统计 (Priority: P4)

管理员登录后进入仪表盘，查看系统概览统计数据（文章总数、已发布数量、
草稿数量、总阅读量）。管理员可以管理所有用户的文章（编辑、删除、
状态变更）、管理分类（增删改查）、管理标签（增删改查，使用中的标签
不可删除）、管理用户角色、管理站点设置（博客名称、关于页内容、社交链接）。
管理员还可以管理图片上传。

**Why this priority**: 管理功能是博客运营的保障，但在内容量和用户量
较小时可手动处理。排在最后是因为在系统初期，作者往往就是唯一管理员，
管理功能的紧迫性最低。

**Independent Test**: 管理员登录后访问仪表盘，确认统计数据准确。
进入用户管理页面，修改某用户的角色为作者，确认该用户获得作者权限。
创建新分类和标签，确认作者可在编辑器中使用。尝试删除已被文章使用
的标签，确认系统阻止并提示。编辑或删除任意作者的文章，确认操作成功。
上传图片，确认图片可在文章编辑器中引用。

**Acceptance Scenarios**:

1. **Given** 管理员已登录，**When** 访问仪表盘页面，**Then** 展示文章总数、已发布文章数、草稿数、总阅读量、用户总数、最近发布的文章列表。
2. **Given** 管理员在文章管理页面，**When** 查看文章列表，**Then** 展示所有作者的文章（不仅是自己的），可以编辑、删除、变更任何文章的状态。
3. **Given** 管理员在分类管理页面，**When** 创建新分类（填写名称和描述），**Then** 分类创建成功，作者可在编辑器中选择。
4. **Given** 管理员在分类管理页面，**When** 编辑或删除一个分类，**Then** 编辑后分类信息更新，删除后分类不再可用（该分类下的文章变为"未分类"）。
5. **Given** 管理员在标签管理页面，**When** 创建新标签（填写名称），**Then** 标签创建成功，作者可在编辑器中选择。
6. **Given** 管理员在标签管理页面，**When** 尝试删除一个被 3 篇文章使用的标签，**Then** 系统拒绝删除并提示"该标签正被 3 篇文章使用，无法删除"。
7. **Given** 管理员在标签管理页面，**When** 删除一个未被任何文章使用的标签，**Then** 标签删除成功。
8. **Given** 管理员在用户管理页面，**When** 将某用户的角色从读者修改为作者，**Then** 该用户下次登录时获得作者权限。
9. **Given** 管理员在图片管理页面，**When** 上传一张图片（支持常见图片格式），**Then** 上传成功，返回可引用的图片 URL。

### Edge Cases

- 文章标题为空时提交，系统拒绝并提示"标题不能为空"。
- 文章内容为空时保存为已发布，系统拒绝并提示"内容不能为空"（草稿状态可允许空内容）。
- 用户注册时使用已被注册的邮箱，系统提示"该邮箱已被注册"。
- 用户注册时使用弱密码（少于 8 位），系统提示密码强度不足。
- 搜索关键词为空时，返回所有文章（等同于浏览全部）。
- 访问不存在的文章详情页，展示 404 页面。
- 文章标题修改后 slug 默认同步更新（基于新标题重新生成），但若作者已手动编辑过 slug，则保持不变。
- 两篇文章标题相同（或生成相同 slug）时，系统自动在 slug 末尾追加随机后缀确保 URL 唯一性。
- 分页超出实际总页数时，重定向到最后一页或显示"没有更多文章"。
- 读者通过直接 URL 访问后台管理页面，系统重定向到首页并提示"无权访问"。
- GitHub OAuth 授权失败或被用户取消，返回登录页面并提示"GitHub 登录失败，请重试"。
- GitHub OAuth 登录时，若 GitHub 主邮箱与已注册邮箱匹配，自动关联账号并合并登录方式；若用户已在设置中手动绑定过 GitHub，则以手动绑定为准。
- 上传图片超过大小限制（默认 10MB），系统拒绝并提示文件过大。
- 读者角色用户尝试访问图片上传功能时，系统返回权限不足错误。
- 作者 A 尝试查看或删除作者 B 上传的图片时，系统拒绝并提示无权操作。
- 多个标签同名（大小写不同）时，系统统一为小写存储，避免重复。
- 作者删除文章后，关联的评论也应一并删除（如后续实现评论功能）。

## Requirements *(mandatory)*

### Functional Requirements

**用户系统**
- **FR-001**: 系统 MUST 支持邮箱和密码注册账号，注册时需提供邮箱、密码（≥8 位）、昵称。
- **FR-002**: 系统 MUST 支持邮箱密码登录，验证失败时给出明确错误提示。
- **FR-003**: 系统 MUST 支持 GitHub OAuth 登录，首次登录时通过邮箱匹配自动关联已有账号（若 GitHub 主邮箱与已注册邮箱一致），无匹配则自动创建新账号。
- **FR-004**: 系统 MUST 定义三种用户角色：管理员（admin）、作者（author）、读者（reader），新注册用户默认为读者。
- **FR-005**: 系统 MUST 在每次请求时验证用户身份和角色权限，不可仅依赖前端界面隐藏。

**文章系统**
- **FR-006**: 系统 MUST 支持 Markdown 格式编写文章内容，并提供实时预览。
- **FR-007**: 每篇文章 MUST 具有以下属性：标题、正文内容、slug（URL 标识符，由标题自动生成，作者可手动编辑）、摘要、封面图 URL、文章状态（draft/published/archived）。
- **FR-007a**: 文章详情页 URL MUST 使用 `/posts/[id]-[slug]` 格式，其中 id 为数据库主键保证唯一性，slug 为可编辑的语义化标识符（中文标题自动转拼音）。
- **FR-008**: 文章 MUST 支持三种状态：草稿（仅作者可见）、已发布（公开可见）、已归档（详情页可访问但不展示在首页列表）。
- **FR-009**: 系统 MUST 支持按关键词搜索已发布文章的标题和正文内容。
- **FR-010**: 文章详情页 MUST 展示并递增阅读量计数。
- **FR-011**: 每篇文章 MUST 支持 SEO 元信息配置：SEO 标题、SEO 描述、OG 图片 URL，未配置时使用文章标题和摘要作为默认值。
- **FR-012**: 作者 MUST 只能编辑和删除自己的文章；管理员可以管理所有文章。

**标签与分类**
- **FR-013**: 系统 MUST 支持标签（Tag）的创建、编辑、删除，标签与文章为多对多关系。
- **FR-014**: 系统 MUST 阻止删除正在被文章使用的标签，并提示关联文章数量。
- **FR-015**: 系统 MUST 支持分类（Category）的创建、编辑、删除，分类与文章为单选关系。
- **FR-016**: 删除分类后，该分类下的文章分类字段置空（未分类状态）。

**前台页面**
- **FR-017**: 首页 MUST 按发布时间倒序展示已发布文章列表，支持分页（每页 20 篇）。
- **FR-018**: 文章详情页 MUST 渲染 Markdown 内容，展示作者信息、发布时间、标签、阅读量。
- **FR-019**: 系统 MUST 提供按标签筛选文章的页面，展示包含指定标签的所有已发布文章。
- **FR-020**: 系统 MUST 提供按分类筛选文章的页面，展示属于指定分类的所有已发布文章。
- **FR-021**: 系统 MUST 提供关于页面，展示博主介绍等内容，内容来自站点设置（管理员可在后台编辑），前台以静态页面形式渲染。
- **FR-021a**: 系统 MUST 提供站点设置功能（仅管理员可访问），支持编辑：博客名称、博主介绍（关于页内容）、社交链接（GitHub、Twitter 等）、博客 Logo。

**后台管理**
- **FR-022**: 仪表盘 MUST 展示：文章总数、已发布数量、草稿数量、总阅读量、用户总数。统计数据采用定时缓存策略，每 5 分钟刷新一次，仪表盘读取缓存值（允许最多 5 分钟的数据延迟）。
- **FR-023**: 文章管理页面 MUST 支持列表展示（含筛选）、新建、编辑、删除、状态切换。
- **FR-024**: 分类管理页面 MUST 支持分类的增删改查。
- **FR-025**: 标签管理页面 MUST 支持标签的增删改查，删除时校验关联文章。
- **FR-026**: 用户管理页面 MUST 支持查看用户列表和修改用户角色（仅管理员可访问）。

**非功能需求**
- **FR-027**: 全部前台页面 MUST 支持响应式布局，在移动端（宽度 ≥ 320px）和桌面端（宽度 ≥ 1280px）均可正常使用。
- **FR-028**: 首页和文章详情页 MUST 使用增量静态再生（ISR）策略，确保内容更新后在合理时间内（≤ 60 秒）对访问者可见。
- **FR-029**: 系统 MUST 支持图片上传（至少支持 JPEG、PNG、WebP 格式），作者和管理员可以上传图片，上传的图片返回可访问的 URL。管理员可以查看和管理所有用户上传的图片，作者只能查看和管理自己上传的图片。
- **FR-030**: 图片存储 MUST 预留云存储扩展接口（如 S3/OSS），当前默认使用本地文件存储。

### Key Entities *(include if feature involves data)*

- **User（用户）**: 系统使用者。关键属性：邮箱（唯一）、加密密码（邮箱注册时可为空，若仅通过 GitHub OAuth 登录）、GitHub ID（可为空，若仅通过邮箱注册）、昵称、角色（admin/author/reader）、注册时间、最后登录时间。一个账号可同时绑定邮箱密码和 GitHub OAuth 两种登录方式。
- **Article（文章）**: 用户创作的博客内容。关键属性：标题、正文（Markdown）、slug（URL 标识符，可编辑）、摘要、封面图 URL、状态（draft/published/archived）、SEO 标题、SEO 描述、OG 图片 URL、阅读量、发布时间、更新时间。关联：作者（多对一）、标签（多对多）、分类（多对一）。
- **Tag（标签）**: 文章的内容标签。关键属性：名称（唯一，大小写不敏感）、创建时间。关联：文章（多对多）。
- **Category（分类）**: 文章的分类目录。关键属性：名称（唯一）、描述、创建时间。关联：文章（一对多）。
- **Media（媒体文件）**: 上传的图片等文件。关键属性：原始文件名、存储路径、访问 URL、文件大小、MIME 类型、上传者、上传时间。
- **SiteConfig（站点设置）**: 博客全局配置（单例）。关键属性：博客名称、博主介绍（关于页内容）、社交链接（JSON 格式，含 GitHub/Twitter 等链接）、博客 Logo URL。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 读者从访问首页到完整阅读一篇文章，整个过程操作步骤不超过 3 步（首页 → 点击文章 → 阅读），时间不超过 30 秒（含页面加载）。
- **SC-002**: 作者从登录到发布一篇新文章（含 Markdown 写作、标签选择、分类选择），熟练用户可在 5 分钟内完成。
- **SC-003**: 首页和文章详情页在桌面端首次加载时间不超过 2 秒（含图片），后续导航（ISR 缓存命中）不超过 500 毫秒。
- **SC-004**: 系统在 100 篇已发布文章、10 个分类、50 个标签的数据规模下，搜索和筛选操作的响应时间不超过 1 秒。
- **SC-005**: 移动端（宽度 ≤ 375px）所有前台页面内容可读、操作可用，无横向滚动条，触摸目标不小于 44×44px。
- **SC-006**: 用户注册到登录成功的转化率达到 95% 以上（排除无效邮箱）。
- **SC-007**: 管理员完成一次完整的分类和标签管理操作（创建或删除）不超过 1 分钟。

## Assumptions

- 系统初期为单用户博客场景（博主即管理员和唯一作者），但架构预留多用户和角色扩展能力。
- 部署环境支持 Node.js 运行时和关系型数据库。
- 域名和 HTTPS 证书已配置（OAuth 回调 URL 和安全的 Cookie 传输依赖 HTTPS）。
- 图片存储使用服务器本地文件系统，云存储为未来增强项。
- 文章搜索为简单的数据库全文搜索（LIKE 或基础全文索引），不引入独立搜索引擎。
- 用户使用的主流浏览器支持现代 Web 标准（Chrome/Firefox/Safari/Edge 最近 2 个大版本）。
- 评论功能虽在角色描述中提到（"读者只能浏览和评论"），但本轮规范不包括完整的评论系统实现——评论仅作为角色权限边界的说明，具体评论功能留待后续特性扩展。
